## v1 Requirements — All Complete

### Foundation (Storage Schema & Identity)

- [x] **FND-01**: Extension assigns a stable UUID to each instruction on first sight; UUID becomes that instruction's permanent identity (renames do not break identity). — **✓ Phase 1** (registry.ts, crypto.randomUUID())
- [x] **FND-02**: Each instruction has an `updated_at` timestamp recorded with every change so per-item conflicts can be resolved by last-write-wins. — **✓ Phase 1** (registry.ts updateItem)
- [x] **FND-03**: Deleted instructions are recorded as soft-delete tombstones with a `deleted_at` timestamp so deletes propagate across devices without being resurrected by stale data on another device. — **✓ Phase 1** (registry.ts deleteItem, applyRemote tombstone win)
- [x] **FND-04**: Storage uses a versioned schema (`schema_version` key) and namespaced keys (`sysins:*`) so future migrations are possible without ambiguity. — **✓ Phase 1** (sysins:meta schemaVersion: 1)
- [x] **FND-05**: Storage layout separates a small registry (UUID → `{title, updated_at, deleted_at}`) from instruction body keys, so merge decisions can be made without fetching every body. — **✓ Phase 1** (sysins:registry + sysins:body:*)
- [x] **FND-06**: All sync state required to resume work (last-pushed snapshot, in-progress flag, pending-merge queue) is persisted in `chrome.storage.local`, never only in service-worker memory. — **✓ Phase 1** (sync-state.ts, LAST_PUSHED_KEY, SYNC_PENDING_KEY)

### Sync Engine — Push (localStorage → chrome.storage.sync)

- [x] **PUSH-01**: When AI Studio writes to `localStorage["aistudio_all_system_instructions"]`, the extension detects the change without requiring a page reload. — **✓ Phase 2** (ls-observer.js MAIN-world patch + 2s polling fallback)
- [x] **PUSH-02**: The extension assigns UUIDs to any instructions that do not yet have one, computes a per-item diff against the last-pushed snapshot, and pushes only the changed items. — **✓ Phase 3** (push-engine.ts diffAndAccumulate)
- [x] **PUSH-03**: All chunks/keys for one push cycle are written via a single `chrome.storage.sync.set({...})` call — never per-item loops — so writes stay under the 120/min and 1800/hr rate limits. — **✓ Phase 3** (alarm-flush.ts flushPendingWrite single batched set)
- [x] **PUSH-04**: Single instructions larger than the 8KB-per-item quota are transparently chunked across multiple `sysins:body:<uuid>:cN` keys; reassembly is deterministic and does not require user action. — **✓ Phase 3** (storage-layout.ts splitIntoChunks/joinChunks)
- [x] **PUSH-05**: A `null`, missing, or empty-array localStorage read is never auto-propagated as "user deleted everything" — empty results are treated as a detection failure unless the user explicitly clicks Push Now. — **✓ Phase 2** (isValidPayload guard in content/index.ts and ls-observer.js)
- [x] **PUSH-06**: Unknown fields on instruction items (anything beyond `title` and `text`) are preserved end-to-end so future AI Studio schema additions are not silently dropped. — **✓ Phase 2** (RawInstruction index signature, verbatim passthrough)
- [x] **PUSH-07**: Pushes are debounced/throttled (target: 30-second alarm flush) so a flurry of edits in AI Studio coalesces into one batched write. — **✓ Phase 3** (alarm-flush.ts scheduleFlush 30s chrome.alarms)

### Sync Engine — Pull (chrome.storage.sync → localStorage)

- [x] **PULL-01**: When `chrome.storage.sync` reports a change from another device, the service worker wakes, computes the merged registry, and applies the result via the content script. — **✓ Phase 4** (pull-engine.ts handleRemoteChanged)
- [x] **PULL-02**: Per-item conflicts are resolved by last-write-wins on `updated_at`; tombstones (`deleted_at > updated_at`) win unconditionally over live items so a delete on one device cannot be resurrected by an older live copy elsewhere. — **✓ Phase 4** (registry.ts applyRemote authoritative timestamp model)
- [x] **PULL-03**: Pulls dispatched into an open AI Studio tab write `localStorage` and dispatch a synthetic `StorageEvent`; if AI Studio's React does not pick it up, the popup/badge surfaces a "Refresh AI Studio to see latest" hint instead of failing silently. — **✓ Phase 4+5** (content/index.ts APPLY_REMOTE + BannerRow.svelte refresh hint)
- [x] **PULL-04**: Pull-initiated writes do not trigger another push (no infinite sync loop) — implemented via a write-suppression flag during apply, plus diff-against-last-pushed. — **✓ Phase 4** (APPLYING_REMOTE_KEY suppression flag + diff check in push-engine)
- [x] **PULL-05**: When two AI Studio tabs are open, only one applies a remote update so concurrent writes do not race. — **✓ Phase 4** (chrome.tabs.query → first matching tab only)

### Bootstrap & Account Safety

- [x] **BOOT-01**: First-install on a device is a union merge between local AI Studio data and remote sync data, never a pull-overwrite — instructions present on either side survive. — **✓ Phase 4** (bootstrap.ts handleLsBootstrap union merge)
- [x] **BOOT-02**: Items found in `localStorage` without a UUID are matched to remote items by title at bootstrap only; once UUIDs are assigned, title-matching is never used again. — **✓ Phase 4** (bootstrap.ts title-match UUID assignment, one-time only)
- [x] **BOOT-03**: Before auto-sync runs, the extension performs an account-mismatch pre-flight check: if the Chrome profile's signed-in account differs from the AI Studio account on the page, auto-sync is paused and the popup surfaces a warning. — **✓ Phase 4** (account-preflight.ts, chrome.identity.getProfileUserInfo, ACCOUNT_MISMATCH error state)

### Popup UI

- [x] **UI-01**: The toolbar icon opens a popup that shows the last successful sync timestamp, current sync state (idle / syncing / error), and a count of synced instructions. — **✓ Phase 5** (StatusHeader.svelte)
- [x] **UI-02**: The popup lists every instruction with its title and per-item `updated_at` timestamp; the list reflects the merged state, not the raw localStorage view. — **✓ Phase 5** (InstructionList.svelte, registry-only query)
- [x] **UI-03**: The popup exposes a "Push now" button that flushes pending writes immediately, bypassing the debounce. — **✓ Phase 5** (ActionRow.svelte + PUSH_NOW handler in index.ts)
- [x] **UI-04**: The popup exposes a "Pull now" button that forces a fresh read from `chrome.storage.sync` and re-applies the merged result. — **✓ Phase 5** (ActionRow.svelte + PULL_NOW handler in index.ts)
- [x] **UI-05**: The popup shows an explicit error state (badge color + human-readable message) for: quota exceeded, sync unavailable, account mismatch, malformed remote payload — no error is silently swallowed. — **✓ Phase 5** (BannerRow.svelte, 9 ErrorState values covered)
- [x] **UI-06**: The toolbar badge reflects sync health at a glance: green = healthy, amber = needs attention (e.g. refresh hint, pending writes), red = error. — **✓ Phase 5** (alarm-flush.ts + pull-engine.ts badge writes, badge clear on pull success)

### Export / Import

- [x] **EXPORT-01**: The popup provides a "Export to JSON" action that produces a single human-readable JSON file containing every instruction (title, text, UUID, `updated_at`). — **✓ Phase 5** (App.svelte exportJSON, batched body fetch, filename date-stamped)
- [x] **EXPORT-02**: The popup provides an "Import from JSON" action that ingests a previously exported file and routes every item through the standard merge path — imported items get UUIDs assigned, conflicts resolve via last-write-wins, tombstones are respected. — **✓ Phase 5** (App.svelte handleFileSelected → IMPORT_ITEMS → importItems() union merge)

### Distribution & Hygiene

- [x] **DIST-01**: The extension is sideloadable as an unpacked build for personal use on the author's machines. — **✓ Phase 1** (WXT scaffold, `npm run build` → .output/chrome-mv3/)
- [x] **DIST-02**: The manifest requests only the minimum permissions required. — **✓ Phase 1** (storage, scripting, alarms, identity — no <all_urls>)
- [x] **DIST-03**: The codebase, manifest, and build output are kept Chrome-Web-Store-clean. — **✓ Phase 1** (manifest snapshot assertion, no debug flags)
- [x] **DIST-04**: No instruction data leaves the user's own Google Chrome sync. — **✓ Phase 1** (static scan for fetch/XMLHttpRequest/WebSocket + ESLint)
