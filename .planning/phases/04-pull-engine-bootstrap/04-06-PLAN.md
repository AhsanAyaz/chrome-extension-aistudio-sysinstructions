  <how-to-verify>
Reload the unpacked extension in Chrome (chrome://extensions → reload). Open the extension's service worker DevTools inspector (SW DevTools console) for each check.

CRITERION 1 — Remote pull propagation (PULL-01):
  Simulate a remote push arriving by running this in the SW DevTools console:
    await chrome.storage.sync.get(null).then(d => console.log(JSON.stringify(d)))
  To simulate a remote change, use fakeBrowser in a test OR manually set a known registry entry:
    await chrome.storage.sync.set({ 'sysins:registry': { 'test-uuid-001': { title: 'Remote Test', updatedAt: Date.now(), deletedAt: null, chunks: 1 } }, 'sysins:body:test-uuid-001:c0': '{"text":"Hello from remote"}' })
  Then open AI Studio and check: does localStorage["aistudio_all_system_instructions"] contain the new item?
  Expected: item appears within 60 seconds (or immediately if the tab is already active).
  PASS if: the instruction appears in AI Studio without a page reload.

CRITERION 2 — Tombstone wins (PULL-02):
  Pre-condition: the item from Criterion 1 exists locally.
  Simulate a remote tombstone: in SW DevTools, run:
    const reg = await chrome.storage.sync.get('sysins:registry');
    reg['sysins:registry']['test-uuid-001'] = { title: 'Remote Test', updatedAt: 1000, deletedAt: Date.now(), chunks: 0 };
    await chrome.storage.sync.set(reg);
  Wait for the pull to propagate (trigger it by calling handleRemoteChanged manually if needed).
  Check AI Studio localStorage: the item should be gone.
  PASS if: tombstoned item no longer appears in AI Studio after pull.

CRITERION 3 — First-install union merge (BOOT-01, BOOT-02):
  Test procedure:
    a. Clear all extension sync storage: chrome.storage.sync.clear() in SW DevTools.
    b. Manually add an item to AI Studio's localStorage in the AI Studio tab console:
       localStorage.setItem('aistudio_all_system_instructions', JSON.stringify([{title:'Local Only',text:'local text'}]))
    c. Simulate the bootstrap trigger: in SW DevTools, run:
       await chrome.storage.local.set({ 'sysins:local:bootstrapNeeded': { triggeredAt: Date.now() } })
    d. Reload the AI Studio tab (triggers content script bootstrap check).
    e. Check chrome.storage.sync.get(null) — should contain the local item with an assigned UUID.
    f. Also add a remote-only item BEFORE step (d): set a registry entry + body key in sync for 'remote-uuid-002'.
    g. After bootstrap merge: check localStorage — should contain BOTH local and remote items.
  PASS if: both local-only and remote-only items survive the bootstrap (neither is overwritten).

CRITERION 4 — No infinite loop (PULL-04):
  Method: count chrome.storage.sync.set() calls after one AI Studio edit.
  In SW DevTools, monkey-patch: const orig = chrome.storage.sync.set.bind(chrome.storage.sync); let count = 0; chrome.storage.sync.set = function(...args) { count++; console.log('sync.set #' + count); return orig(...args); };
  Then make one change in AI Studio and wait 35 seconds.
  PASS if: exactly 1 chrome.storage.sync.set() call fires (the push flush). If a second fires shortly after (the pull triggering a re-push), that is a FAIL.

CRITERION 5 — Single-tab deduplication (PULL-05):
  Open two AI Studio tabs (both on aistudio.google.com).
  In SW DevTools, trigger a fake remote change (same as Criterion 1).
  Open DevTools Console in BOTH AI Studio tabs and filter by '[sysins]'.
  The content script logs '[sysins] applyRemoteLocally called <N>' each time it applies a remote update
  (added in plan 04-05 Task 4 inside the applyRemoteLocally function).
  PASS if: only ONE tab console shows the '[sysins] applyRemoteLocally called' log line.
  The other tab should show no such log — confirming SW sent APPLY_REMOTE to only the active tab.

CRITERION 6 — Account mismatch detection (BOOT-03):
  In SW DevTools, call:
    chrome.identity.getProfileUserInfo({ accountStatus: 'ANY' }, (info) => console.log(info));
  Note the chrome profile email.
  Then call the account preflight manually:
    // Import path via eval trick (MV3 SW has module scope)
    // OR: check that setErrorState('ACCOUNT_MISMATCH') sets the syncStatus correctly
    const status = await chrome.storage.local.get('sysins:local:syncStatus');
    console.log(status);
  To trigger ACCOUNT_MISMATCH manually: in SW DevTools run:
    await chrome.storage.local.set({ 'sysins:local:syncStatus': { state: 'error', lastSyncAt: 0, errorState: 'ACCOUNT_MISMATCH' }})
    const status = await chrome.storage.local.get('sysins:local:syncStatus');
    console.log(status); // should show ACCOUNT_MISMATCH
  PASS if: ACCOUNT_MISMATCH state is settable and readable, and the account-preflight.ts function exists (confirmed by build).
  Note: Full end-to-end BOOT-03 with actual mismatched accounts requires two Google accounts — if only one account is available, confirm the function exists and would fire correctly based on code review.
